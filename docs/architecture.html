01System Overview

A document-grounded financial assistant that runs entirely on your machine. Documents are dropped into a folder (or uploaded), then extracted, chunked, embedded and stored in Postgres + pgvector. Each question is handled by a multi-agent LangGraph workflow — a planner routes the request and classifies its complexity, a retriever pulls grounding context with hybrid search, and an analyst writes the answer (streamed token-by-token). A data-driven model router picks which local model writes the answer — fast for simple questions, strong for complex ones — using a policy measured by the eval harness. Everything is served locally by Ollama; no cloud account or calls are required.

02The Agent Graph (LangGraph)

The heart of the platform. A compiled StateGraph passes a shared AgentState between specialized nodes. The planner conditionally routes to retrieval for document questions, or straight to the analyst for general-knowledge ones. The same planner call also labels the query SIMPLE/COMPLEX, which the model router uses to choose the analyst's model — agentic behavior with no extra latency.

🔀 Model router: the Planner's SIMPLE/COMPLEX label selects the Analyst's model from the eval-derived policy. The current measured policy is counterintuitive — simple → qwen2.5:3b (strong), complex → llama3.2:1b (fast) — because the harness found the strong model only earns its keep on short definitional answers. The Planner can also skip the Retriever and go straight to the Analyst for general-knowledge questions. Streaming runs on the caller's thread so tokens reach the Streamlit UI live.

03Model Router & Evals

Different local models trade quality for speed. Rather than guess, the platform measures each model with an offline eval harness, derives a routing policy, and the router applies it per query. The harness needs only Ollama (no Postgres). See the evals deep dive for the dataset, metrics and latest results.
