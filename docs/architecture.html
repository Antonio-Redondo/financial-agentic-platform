01System Overview

A document-grounded financial assistant that runs entirely on your machine. Documents are dropped into a folder (or uploaded), then extracted, chunked, embedded and stored in Postgres + pgvector. Each question is handled by a multi-agent LangGraph workflow — a planner routes the request and classifies its complexity, a retriever pulls grounding context with hybrid search, and an analyst writes the answer (streamed token-by-token). A data-driven model router picks which local model writes the answer — fast for simple questions, strong for complex ones — using a policy measured by the eval harness. A guardrails layer validates input, neutralizes prompt-injection, and redacts PII (SSN, bank account, card, routing, email…) at the input, context and output boundaries, while a semantic response cache and a pgvector connection pool cut latency on repeat questions. Everything is served locally by Ollama; no cloud account or calls are required.

🔀 Model router: the Planner's SIMPLE/COMPLEX label selects the Analyst's model from the eval-derived policy. The current measured policy is counterintuitive — simple → qwen2.5:3b (strong), complex → llama3.2:1b (fast) — because the harness found the strong model only earns its keep on short definitional answers. The Planner can also skip the Retriever and go straight to the Analyst for general-knowledge questions. Streaming runs on the caller's thread so tokens reach the Streamlit UI live.
🛡️ Guardrails wrap the graph: the input is validated (length, prompt-injection) and the query + history are PII-redacted before any model call (so nothing raw reaches the planner, analyst, or an off-machine trace); retrieved context is scanned for indirect injection and PII before it reaches the Analyst; the Analyst's answer is PII-redacted in the live token stream and again in Finalize before it's shown or stored. ⚡ A semantic cache sits in front of the whole graph — an exact or near-duplicate standalone question returns the prior answer and skips planning, retrieval and generation entirely.

06Security & Guardrails (PII)

A local, dependency-free guardrails layer (src/guardrails/) enforced at the three I/O boundaries of the agent graph. It defends against the relevant OWASP-LLM risks — prompt injection (direct & indirect), sensitive-information disclosure, and unbounded input — and redacts personally identifiable / financial data so it never reaches the screen, the logs, or an off-machine trace. Everything is env-tunable and fails open (a guardrail error never blocks a query).

🔢 PII detectors

  • SSN, EIN, credit card (Luhn), IBAN (mod-97), ABA routing (checksum)
  • Bank account, email, phone, IP
  • Checksums + context-keyword gating cut false positives

🎭 Masking strategies

  • partial — keep last 4 (e.g. •••-••-1234)
  • label / hash / remove variants
  • GUARDRAILS_PII_STRATEGY

🧱 Injection defense

  • Heuristics: override / role-reset / jailbreak / exfil / delimiter-spoof
  • Direct: flag or block the query
  • Indirect: defang instruction-like lines in retrieved docs

⚙️ Posture & config

  • Prompt / output / stream / context / log redaction on by default
  • Ingest redaction off by default (detect + tag in metadata)
  • GUARDRAILS_* in .env; master GUARDRAILS_ENABLED

With GUARDRAILS_PII_REDACT_PROMPTS=true (default) the query and chat history are masked before any model call, so even with LangSmith tracing on, traces carry no raw PII; embeddings run locally on Ollama. Disable tracing for a fully air-gapped posture, and set GUARDRAILS_PII_REDACT_ON_INGEST=true to keep raw PII out of pgvector at rest.

07Performance — Caching & Connection Pool

Two layers shave latency without touching answer quality: a semantic response cache that returns a prior answer for repeated or near-duplicate questions, and a pgvector connection pool that reuses warm database connections instead of opening a fresh one per query.

⚡ Semantic response cache

src/agents/semantic_cache.py

  • Exact hits are O(1) (normalized text) — no embedding call
  • Semantic hits match on query-embedding cosine ≥ RESPONSE_CACHE_THRESHOLD (0.97)
  • Bounded LRU (RESPONSE_CACHE_MAX) + optional TTL
  • Self-invalidates: namespaced by model + chunk_count; cleared on re-ingest
  • Only standalone questions cached (history / filters bypass it)

🔌 Connection pool

src/storage/db.py · psycopg-pool

  • Process-wide pool; register_vector per connection
  • DB_POOL_MIN_SIZE / MAX_SIZE / TIMEOUT / MAX_IDLE
  • get_conn() API unchanged across all call sites
  • Graceful fallback to direct connections if unavailable

🟢 Existing reuse caches

embeddings.py · graph.py · router.py

  • @lru_cache embedder & per-model analyst clients
  • Eval policy + guardrail config read once per process
  • Content-hash dedupe skips re-embedding unchanged docs

📊 Observable

  • Sidebar shows cache size + hit-rate; clear button
  • Cached turns carry a ⚡ cached chip + similarity
  • Trace records the cache kind (exact / semantic)

08Cost Optimization

Because the platform runs 100% locally on Ollama, there is no per-token API bill — "cost" here is compute, latency and tokens on your own CPU/GPU. The design attacks that on five fronts: skip work entirely (cache), bound the work (token budgets + input cap), right-size the model (complexity router), reuse warm resources (pool + client caches), and keep the expensive retrieval tricks off by default. An offline eval harness measures latency so the routing policy is tuned from real numbers, not guesses.

⚡ Skip work entirely

semantic_cache.py

  • Exact hit → O(1), not even an embedding call
  • Semantic hit (cosine ≥ 0.97) returns the prior answer — zero LLM tokens
  • A hit skips plan + retrieve + generate — the whole graph
  • Highest-leverage lever for repeated / near-duplicate questions

✂️ Bound the work

graph.py · llm.py · guardrails/

  • Per-agent num_predict caps: planner 24, analyst 700, judge 8
  • Input char cap GUARDRAILS_MAX_INPUT_CHARS (8000) rejects oversized prompts before any model call
  • PII redaction also trims raw spans out of prompts
  • Chunk size 500/100 keeps embed + context small

🔀 Right-size the model

router.py · evals/

  • Complexity label (already produced by the planner) → fast vs strong model: no extra latency to decide
  • Policy from policy.json, gated by QUALITY_TOLERANCE — only pay for the strong model when its judge gain clears the bar
  • Expensive retrieval (multi-query, HyDE, LLM rerank) off by default — each adds 1–3 extra LLM calls

🔌 Reuse warm resources

db.py · embeddings.py · graph.py

  • psycopg-pool reuses warm connections (no per-query TCP + auth + adapter setup)
  • @lru_cache embedder + per-model analyst clients — built once
  • Content-hash dedupe skips re-embedding unchanged docs
  • Lazy load: the strong model only loads on the first complex query

⏱️ Measure to tune

evals/run_evals.py

  • Wall-clock latency (mean / p50 / p95) per model per complexity
  • fast = min latency, strong = max judge; route picks fast unless Δjudge > tolerance
  • Re-run python -m evals.run_evals to refresh policy.json when models change

Current policy is a measured no-op for routing. The latest evals/results/policy.json routes both simple and complex to qwen2.5:3b (Δjudge −0.50 and +0.00, both within the 0.4 tolerance), so the planner still spends one short classification call but the analyst model never actually switches. The router is a working, cost-aware framework that currently finds no speed win to take — swap in models with a wider speed/quality spread and re-run the evals, and it will start saving latency automatically.

14Developer Agents & Skills (Copilot)

Outside the runtime, the repo also ships a set of Copilot subagents and skills in .github/ that codify how to work on this codebase. Agents are personas with a mission, responsibilities and an output template (invoked through the chat runSubagent flow). Skills are procedural companions that an agent — or the default assistant — reads when a matching task comes up. Most agents have a paired skill: the agent defines who and what, the skill walks the step-by-step how for this specific stack (pgvector + Ollama + LangGraph, Windows + .venv).
