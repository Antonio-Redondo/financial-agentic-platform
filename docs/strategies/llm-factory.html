🟢 LLM Factory

A single helper, build_llm(), is the only place the platform constructs a ChatOllama client — so the local model, endpoint and sampling defaults are configured once. Callers pass a per-agent token budget and temperature, keeping a tiny router classifier and a long-form writer on the same factory.

src/agents/llm.py single ChatOllama factory per-agent token budgets

One place to build a client

Every agent node in the LangGraph workflow needs an LLM, but none of them talk to langchain_ollama directly. They all go through one function so the Ollama model name, base URL and shared sampling knobs live in exactly one file. 

# src/agents/llm.py — the whole module
import os
from langchain_ollama import ChatOllama

def build_llm(num_predict: int = 1024, temperature: float = 0.7,
              model: str = None) -> ChatOllama:
    return ChatOllama(
        model=model or os.getenv("OLLAMA_MODEL", "llama3.2:1b"),
        base_url=os.getenv("OLLAMA_BASE_URL", "http://localhost:11434"),
        temperature=temperature,
        top_p=0.9,
        num_predict=num_predict,
    )

That's the entire file. It is a thin constructor — no caching, no global state — which is why it's safe to call from any node.

The explicit model argument

build_llm() takes an optional model name. When omitted it falls back to the $OLLAMA_MODEL environment variable (and finally to llama3.2:1b), which is the manual default. Passing an explicit model is what lets the platform run several differently-sized clients side by side — e.g. a small classifier and a larger writer — without each caller re-reading the environment.

♻️ Where reuse actually happens

build_llm() itself is not memoised — there is no @lru_cache in this file. Client reuse is handled by the caller in graph.py: the default planner_llm / analyst_llm are built once in _build_llms(), and router-selected analyst models are kept in a _analyst_cache dict so each model name is constructed only once.

Shared defaults vs. per-call settings

Two sampling values are baked into the factory and are the same for every client; two are arguments that each agent sets for itself.

SettingSourceValue
top_pFixed in build_llm()0.9 for all clients
base_url$OLLAMA_BASE_URLdefault http://localhost:11434
modelarg, else $OLLAMA_MODELdefault llama3.2:1b
num_predictarg (default 1024)per-agent — see below
temperaturearg (default 0.7)per-agent — see below

num_predict is documented in the factory's docstring as "max tokens to generate (small for routing/critique agents, large for the writing agent)"; temperature as "0.0 for deterministic routing/review".

Per-agent token budgets & temperatures

The factory only sets the defaults (num_predict=1024, temperature=0.7). The actual per-agent budgets are chosen by the callers in src/agents/graph.py, where each node calls build_llm() with its own values:

Agent / usenum_predicttemperatureWhy
Planner (router classifier)240.0tiny classifier; deterministic routing
Analyst (writes the answer)7000.7modest cap so one answer can't run away on slow CPU

📍 These values are not in llm.py

llm.py defines no agent-specific constants. The numbers above come straight from the build_llm(...) call sites in src/agents/graph.py (_build_llms(), _analyst_for() and the suggestion path). The factory's own defaults of 1024 / 0.7 apply only if a caller omits the arguments.

Base URL & environment knobs

The factory reads exactly two environment variables. Both have sensible local defaults, so the app runs against a stock local Ollama install with no configuration.

OLLAMA_MODEL
fallback model when build_llm() is called without a model arg — default llama3.2:1b
OLLAMA_BASE_URL
Ollama server endpoint — default http://localhost:11434

🌐 Pointing at a remote Ollama

To use a model server on another host, set OLLAMA_BASE_URL (e.g. to a LAN address or a container's host). Because every client is built through this one factory, that single variable redirects planner, analyst and suggestion calls alike — no per-agent override needed.
