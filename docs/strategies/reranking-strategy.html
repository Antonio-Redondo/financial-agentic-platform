Strategy · Precision

🎯 Re-ranking Strategy

Retrieval is tuned for recall — fetch many plausible chunks. Re-ranking adds a precision pass: an LLM acts as a cross-encoder, reading each candidate against the question and keeping only the genuinely relevant few.

src/agents/retrieval.py rerank_llm() off by default

Why re-rank at all?

🧠 Bi-encoder recall vs. cross-encoder precision

Vector and keyword search are bi-encoders: the query and each chunk are encoded separately, then compared. That's fast and scales to the whole corpus, but the model never sees the query and a passage together, so subtle relevance is lost. A cross-encoder reads the (question, passage) pair jointly and judges them in context — far more accurate, but too slow to run over every chunk. The fix: retrieve broadly with the cheap bi-encoders, then re-rank a small shortlist with the expensive cross-encoder.

This platform has no separate cross-encoder model — it reuses the local LLM as one, asking it to score the shortlist. No extra dependency, no extra model to pull.

When RETRIEVAL_RERANK=true, the retriever over-fetches to fetch_k instead of k, then calls rerank_llm() to trim back to the best RETRIEVAL_K. The whole shortlist is scored in a single prompt — one LLM round-trip, not one per chunk — which keeps it affordable on CPU. 

# retrieval.py — rerank_llm(): one batched prompt for the whole shortlist
candidates = results[:20]                 # cap — long prompts kill CPU latency
snippet = (r["content"] or "")[:600]      # trim each passage
prompt = (
  "You are scoring how well each PASSAGE answers the QUESTION. "
  'Reply with ONLY a JSON array of objects `[{"id": <int>, "score": <0-10>}]`' ...
)
# scores normalized to 0–1 and written back as relevance_score; top_k kept
r["relevance_score"] = max(0.0, score / 10.0)

Built to fail safe

  • Parse-proof. If the LLM returns malformed JSON or no scores, the original retrieval order is preserved (results[:top_k]) — re-ranking never worsens results.
  • Bounded cost. The batch is capped at 20 candidates and each passage truncated to 600 chars, so the prompt can't balloon on a large corpus.
  • Single round-trip. Scoring the whole shortlist in one prompt (vs. one call per chunk) is what makes an LLM re-ranker viable on a local model.
  • Score normalization. 0–10 judgements are mapped to a 0–1 relevance_score so the cited context the analyst sees stays on the same scale as the other retrieval modes.

When to turn it on

SituationRecommendation
Fast, interactive CPU chatLeave off — hybrid + RRF is usually enough
Large/noisy corpus, near-duplicate chunksTurn on — precision gain outweighs the extra call
High-stakes, accuracy-critical answersTurn on, ideally paired with multi-query for recall first
GPU availableTurn on — the added latency is small

⚠️ It's the most expensive toggle

Re-ranking adds one large prompt (up to ~20 passages × 600 chars) per query on top of retrieval. On a slow CPU that can dominate response time — which is exactly why it ships disabled and is best combined with the model router so the heavier work lands on the strong model only when warranted.

Configuration knobs

RETRIEVAL_RERANK
false — set true to enable the LLM cross-encoder pass
RETRIEVAL_FETCH_K
20 — shortlist size fed into re-ranking
RETRIEVAL_K
6 — final count kept after re-ranking

The re-ranker runs on the planner/fast model (self.planner_llm) — deterministic and already loaded — so enabling it doesn't pull a second model into memory.
