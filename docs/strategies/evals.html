Strategy · Measurement

⚖️ Evals & Results

Model selection is earned, not guessed. An offline harness answers a fixed set of financial questions with each candidate model, grades them on quality and speed, and derives the router policy from the numbers. Below is exactly how it works — and the results from the latest run.

evals/run_evals.py Ollama only · no Postgres 20-query dataset → policy.json

Why an eval harness?

A 1B model is fast but shallow; a 3B model is slower but sharper — but by how much, and on which kinds of question? Guessing wastes either latency or quality. The harness turns that trade-off into data: it runs the real analyst prompt (imported from graph.analyst_prompt, so the benchmark matches production) and measures each model head-to-head. Crucially it needs only Ollama — it bypasses retrieval and Postgres — so model choice can be re-measured anytime, and it is never run at startup (a full pass takes many minutes).

How answers are graded

Two quality signals plus a cost signal, captured per query.

📐 Key-point coverage — deterministic

The fraction of expected key points found in the answer as normalized substrings (metrics.key_point_coverage). No model in the loop, so it's stable and repeatable — it anchors the noisier judge. Phrasing-tolerant by design (key points are stems like "securit", "refinanc").

⚖️ LLM-as-judge — 1 to 5

The strongest local model scores each answer 1–5 for correctness and relevance against the question and its key points (judge.py). The prompt is deliberately strict and asks for only a single integer; an empty answer scores 1, a parse failure defaults to 3. 

# judge.py — the rubric the judge model is held to
"  5 = fully correct, directly answers, no errors\n"
"  4 = mostly correct and relevant, minor gaps\n"
"  3 = partially correct or incomplete\n"
"  2 = largely wrong or off-topic\n"
"  1 = incorrect, irrelevant, or empty\n"

⏱️ Latency — cost

Wall-clock seconds per answer (mean / p50 / p95), timed through the same ChatOllama path the app uses, so the numbers reflect real CPU behavior.

How the policy is derived

🗳️ Prefer cheap — unless quality clearly justifies the wait

derive_policy() picks fast = lowest mean latency and strong = highest mean judge. Then, per complexity bucket, it routes to the fast model unless the strong model's judge advantage on that bucket exceeds QUALITY_TOLERANCE (default 0.4 on the 1–5 scale). The result is a route map — not a blanket "use the big model for hard things." 

# run_evals.py — derive_policy(), per complexity cx
jf, js = mean_judge(fast, cx), mean_judge(strong, cx)
if (js - jf) <= tolerance:  route[cx] = fast     # quality gain too small → save time
else:                      route[cx] = strong   # quality gain worth the latency

🔬 The counterintuitive — and correct — outcome

The naïve assumption is "small model for simple, big model for complex." The data says the opposite:

  • Simple → qwen2.5:3b (strong). Its judge gain is large (3.60 vs 2.50, Δ +1.10 > 0.4) and it's actually faster here (58s vs 65s). Strictly better — easy call.
  • Complex → llama3.2:1b (fast). On hard questions the strong model's edge evaporates (2.70 vs 2.60, Δ +0.10 ≤ 0.4) while costing nearly 2× the latency (123s vs 63s). Paying double for a rounding-error of quality isn't worth it — keep the cheap model.

Both 3B and 1B struggle on the complex set (neither clears judge 3 / coverage ~0.6), so there's no quality to buy there. This is exactly why the policy is measured, not assumed — and why the router reads it from policy.json instead of a hardcoded rule.

Run it yourself

# both models must be pulled first
ollama pull llama3.2:1b && ollama pull qwen2.5:3b

python -m evals.run_evals                       # full 20-query run
python -m evals.run_evals --limit 5             # quick smoke run
python -m evals.run_evals \
  --models llama3.2:1b,qwen2.5:3b \
  --judge qwen2.5:3b --num-predict 320 \
  --quality-tolerance 0.4

Caveats & how to read these numbers

  • The judge is a small local model. A 3B judge is noisy and can favor verbosity — which is exactly why deterministic coverage is reported alongside it as an anchor.
  • Latency is CPU- and machine-dependent. Absolute seconds reflect this run's hardware; the relative gap between models is the durable signal.
  • One run, n=10 per bucket. Small samples — a borderline Δjudge near the tolerance can flip between runs. Re-run before trusting a close call.
  • The judge can grade itself. Here qwen2.5:3b judges its own answers; treat its absolute scores with appropriate skepticism.
  • Tune the tolerance to your priorities. Lower it to favor quality, raise it to favor speed — the whole policy pivots on that one knob.
