🔀 Model Routing Strategy

Different local models trade quality for speed. Rather than guess, the platform measures each model offline, derives a routing policy, and applies it per query — fast model for simple questions, strong model for complex ones, at no extra latency.

src/agents/router.py evals/ enabled by default

The decision, made for free

The planner already classifies each query SIMPLE or COMPLEX in the same call that decides routing — so complexity is known before the analyst runs, with zero extra LLM round-trips. The router maps that label to a model.

# router.py — pick_model(): eval policy first, else complex→strong / else→fast
def pick_model(complexity):
    policy = load_policy()
    if policy and isinstance(policy.get("route"), dict):
        chosen = policy["route"].get(complexity)
        if chosen: return chosen
    return strong_model() if complexity == "complex" else fast_model()

Policy resolution order

The router resolves the fast/strong pair and the route map from the first source that provides them:

#SourceSet by
1evals/results/policy.jsonMeasured by python -m evals.run_evals — wins when present
2ROUTER_FAST_MODEL / ROUTER_STRONG_MODELManual .env override
3llama3.2:1b / qwen2.5:3bHardcoded defaults

The whole feature is gated by MODEL_ROUTER_ENABLED (default true) and can be toggled live from the Streamlit sidebar; the agent trace shows which model actually wrote each answer. When routing is off, a single manually-chosen model is used everywhere.

How the policy is earned — the eval harness

The route map isn't hand-written; it's the output of an offline benchmark. The harness needs only Ollama (no Postgres), so model selection can be re-measured anytime.

What gets measured

📐 Quality

  • Key-point coverage — deterministic: the fraction of expected key points present in the answer. Anchors the noisier judge signal.
  • LLM-as-judge — the strongest local model rates each answer 1–5 for correctness and relevance.

⏱️ Cost

  • Wall-clock latency per answer (mean / p50 / p95), measured through the same ChatOllama path the live app uses — so the numbers reflect real behavior.

The policy rule

🗳️ Prefer fast — unless quality clearly justifies the wait

Per complexity bucket, the harness picks the fast model by default, switching to the strong model only when its judge-score gain exceeds QUALITY_TOLERANCE. Simple questions (definitions, lookups) almost never clear that bar; multi-step reasoning usually does. The result is written to policy.json with a human-readable rationale.

ComplexityExampleDefault routed modelWhy
simple"What is CPR in mortgage finance?"llama3.2:1b (fast)Definitions don't need the strong model
complex"Stress-test this portfolio for rate risk."qwen2.5:3b (strong)Multi-step reasoning rewards quality

This is the code fallback used when no policy file exists. The measured policy can differ — and on the latest run it actually inverts this (simple → strong, complex → fast), because the strong model's quality edge showed up on simple queries, not complex ones. See the evals & results page for the numbers behind that.

Run it yourself

# both models must be pulled first
ollama pull llama3.2:1b && ollama pull qwen2.5:3b

# measure quality vs latency and (re)derive the policy
python -m evals.run_evals            # full run
python -m evals.run_evals --limit 5  # quick smoke run

# outputs: summary.md · results_<ts>.json · policy.json (picked up live)

🔌 Model-agnostic by design

Every LLM is built through src/agents/llm.py, so swapping or adding models — and changing the router's fast/strong pair — is a config change. The agent topology never moves.
