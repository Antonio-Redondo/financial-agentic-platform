Internals · UI

🎨 Streamlit App

The whole front end is one file: a chat UI that streams the agent's answer token-by-token, a tabbed sidebar for documents / model / search, and a live metrics strip. It talks to a single FinancialAgent held in st.session_state, so model switches and uploads survive Streamlit's rerun-on-every-interaction model without losing your documents.

The chat interface & token streaming

handle_prompt(prompt) is the heart of the UI. It appends the user turn, then opens an assistant st.chat_message with an empty st.empty() placeholder that gets rewritten on every token.

Streaming is driven by an on_token closure passed into the agent. Each token is appended to a mutable streamed["text"] dict and the placeholder is redrawn with a trailing ▌ cursor; once the call returns, the placeholder is replaced with the final answer. 

# app.py — handle_prompt(): the streaming callback path
placeholder = st.empty()
streamed = {"text": ""}

def on_token(tok):
    streamed["text"] += tok
    placeholder.markdown(streamed["text"] + " ▌")   # live cursor

history = st.session_state.messages[:-1]              # prior turns for follow-up rewrite
with st.spinner("🧭 Analyzing your question…"):
    response = st.session_state.agent.process_query(
        prompt, on_token=on_token, history=history)
answer = response.get("output", "No response generated")
placeholder.markdown(answer)

The prior turns (messages[:-1], i.e. excluding the just-appended question) are handed to the agent as history so the conversational rewrite can resolve follow-up phrasing. Any structured response["data"] is rendered afterwards by create_data_visualization() as a line chart / table.

↑/↓ history recall in the chat box

Streamlit's st.chat_input has no built-in history, so inject_history_navigation() injects a small client-side script via components.html(...). It collects the user's previously sent questions, serialises them with json.dumps(history), and binds a keydown listener to the chat textarea.

  • ArrowUp (only when the cursor is at the start of the box) walks back through prior questions.
  • ArrowDown (only at the end of the box) walks forward; going past the newest entry clears the box.
  • The script reaches into window.parent.document to find [data-testid="stChatInput"] textarea, and retries on a 100 ms timer (up to 40 tries) until the element exists.

🪟 Why an injected script

The component is rendered with height=0 — it draws nothing, it only attaches keyboard behaviour. Index state is kept on window.__chatHistIdx so it persists across reruns, and is reset whenever the history length changes.

The metrics dashboard

render_metrics() draws a four-column KPI strip in bordered containers. The key detail is when it runs: main() reserves the slot with metrics_placeholder = st.empty() before the sidebar/chat, then fills it after they have executed — so the counts reflect a just-processed document or new message from this run.

initialize_session_state() constructs the FinancialAgent() exactly once and stashes it in st.session_state.agent. If agent.last_ingestion reports new or errored files from the startup drop-folder scan, it surfaces an st.toast summary.

🔁 One agent across reruns

Streamlit re-executes the whole script on every interaction. Because the agent (and its vector store, cache and LLM clients) lives in st.session_state, switching model or toggling routing rebuilds only the LLM clients via set_model() / set_router() — your uploaded documents and chat history are preserved.

⚠️ Streaming under the rerun model

Tokens are painted by mutating a single st.empty() placeholder from the on_token callback — there is no background thread in handle_prompt. The call runs inside an st.spinner, and a suggestion-button click is queued via st.session_state.pending_prompt and replayed on the next rerun so it goes through the same path as typed input. After processing, st.rerun() redraws the finished conversation.
