🕸️ The Agent Graph (LangGraph)

A lean multi-agent workflow compiled once from a LangGraph StateGraph. Four nodes — planner, retriever, analyst, finalize — pass a shared AgentState dict between them, with the planner deciding both which path the query takes and which model writes the answer. It's tuned for fast local CPU inference.

src/agents/graph.py LangGraph StateGraph planner → retriever → analyst → finalize

The compiled graph & shared state

FinancialGraph._build() assembles a StateGraph(AgentState) with four nodes, a single conditional edge after the planner, and a straight run to END. The compiled app is stored once on self.app; nodes are bound methods that read self.*_llm at call time, so swapping models later never requires recompiling. 

# graph.py — _build()
g = StateGraph(AgentState)
g.add_node("planner", self.planner_node)
g.add_node("retrieve", self.retrieve_node)
g.add_node("analyst", self.analyst_node)
g.add_node("finalize", self.finalize_node)

g.set_entry_point("planner")
g.add_conditional_edges("planner", self._after_plan,
                        {"retrieve": "retrieve", "analyst": "analyst"})
g.add_edge("retrieve", "analyst")
g.add_edge("analyst", "finalize")
g.add_edge("finalize", END)
return g.compile()

Everything the nodes share rides on one TypedDict, AgentState (total=False, so every key is optional and nodes return partial updates that LangGraph merges): 

class AgentState(TypedDict, total=False):
    query: str
    search_query: str       # query after conversational rewrite (if any)
    history: List[Dict]     # prior chat turns: [{role, content}, ...]
    filters: Dict           # optional metadata filters for retrieval
    route: str              # "document" | "general"
    complexity: str         # "simple" | "complex" (drives the model router)
    analyst_model: str      # model chosen to write the answer
    documents: List[Dict]
    context: str
    analysis: str
    answer: str
    log: List[str]          # human-readable trace of what each agent did

The four nodes

Each node takes the AgentState and returns only the keys it changed. Every step also appends a human-readable line to log via the _emit() helper, which both prints to stdout and returns the string so it lands in the trace.

NodeWhat it does
planner_nodeRuns the conversational rewrite (when history is present), then one classification call to set route (document/general) and complexity (simple/complex). Applies the _DOCUMENT_HINTS and _COMPLEX_HINTS backstops. Writes route, complexity, search_query.
retrieve_nodeTool node. Pulls chunks from the VectorStore (optional multi-query/HyDE expansion and LLM rerank), de-duplicates on the first 100 chars, builds the context block, and runs guardrails.sanitize_context() over it. Writes documents, context.
analyst_nodePicks the model via _route_analyst_model() and writes the answer with _analyst_prompt(state), grounded in context when present. Writes analysis, analyst_model.
finalize_nodeOutput guardrail. Runs guardrails.filter_output() to redact PII the model may have surfaced, then writes the cleaned answer.

🧭 One call, two axes

The planner gets both decisions from a single classifier call: it prompts for "EXACTLY two words" like GENERAL SIMPLE or DOCUMENT COMPLEX, then parses route = "document" if "DOCUMENT" in raw else "general" and complexity = "complex" if "COMPLEX" in raw else "simple". The planner LLM is built with num_predict=24, temperature=0.0 — it only needs two words.

Conditional routing & the model router

The only branch in the graph is the conditional edge after the planner, decided by the static helper _after_plan: 

@staticmethod
def _after_plan(state: AgentState) -> str:
    return "retrieve" if state.get("route") == "document" else "analyst"

The complexity label is a separate axis: it feeds the model router rather than the graph topology. _route_analyst_model() turns it into the analyst's model — when routing is on, router.pick_model(complexity) chooses fast vs strong; when off, the manual self.model (default OLLAMA_MODEL, llama3.2:1b) is used. 

def _route_analyst_model(self, state: AgentState) -> str:
    if self.use_router:
        return router.pick_model(state.get("complexity") or "simple")
    return self.model

🛟 Backstops over the small planner

Tiny local models route imperfectly, so the planner adds two safety nets after the LLM call: any query matching _DOCUMENT_HINTS (e.g. document, resume, summarize) is forced to the document route when documents are actually indexed; and a simple label is upgraded to complex if the query has more than 30 words or matches _COMPLEX_HINTS (e.g. analyze, compare, forecast).

Streaming on the caller's thread

run() is the entry point. With no on_token callback it invokes the compiled LangGraph (self.app.invoke({...})). With a callback it delegates to stream_run(), which re-runs the identical steps by hand — planner → (retrieve) → analyst → finalize — on the caller's thread.

🧵 Why not just stream the compiled graph?

LangGraph executes nodes inside a worker thread pool, but Streamlit can only render from the main thread — calling st.* off-thread raises "missing ScriptRunContext" and silently drops the update. Driving the same steps in stream_run on the caller's thread lets the live token callback reach the UI. The non-streaming run() path still uses the compiled LangGraph.

How guardrails wrap the run

The graph holds a guardrails instance from get_guardrails() and applies it at three boundaries, so neither retrieved documents nor the model's output can leak injection or PII:

  • Logged queries. The planner scrubs the query with guardrails.scrub_for_log() before printing/tracing, so server logs never carry raw SSNs, account numbers, etc.
  • Retrieved context. retrieve_node runs guardrails.sanitize_context() to neutralize indirect prompt-injection smuggled in via uploaded documents and mask PII before the context reaches the analyst prompt.
  • Streamed & stored output. The live stream is wrapped by guardrails.output_stream() so PII is masked in the tokens themselves, and finalize_node calls guardrails.filter_output() to redact the final stored answer.

Routing config knobs read at call time: OLLAMA_MODEL (default llama3.2:1b) for the manual model, plus the retriever's RETRIEVAL_K (6), RETRIEVAL_FETCH_K (20) and RETRIEVAL_MODE (hybrid).
