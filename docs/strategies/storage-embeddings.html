🗄️ Storage & Embeddings

The retrieval layer is plain Postgres with the pgvector extension, driven over raw psycopg3. A process-wide connection pool serves pgvector-ready connections, the schema bootstraps itself on first use, and embeddings come from a local Ollama model — so the whole stack stays self-hosted with no per-call setup cost.

Schema bootstrap

ensure_schema() creates the vector extension and two tables, then guards against accidentally crossing embedding models. It's safe to call repeatedly — a module-level _SCHEMA_READY flag short-circuits every call after the first in a process.

  • documents — one row per source file, keyed by a unique source_path, with filename, content_hash, file_size, file_extension, an uploaded_at timestamp and a metadata JSONB column.
  • document_chunks — the searchable units. Each carries a vector({dim}) embedding, the chunk content, a chunk_index, and a document_id foreign key with ON DELETE CASCADE (deleting a document drops its chunks).
  • content_tsv — a generated tsvector column (to_tsvector('english', content), STORED) added via ADD COLUMN IF NOT EXISTS so it backfills existing tables in place. It powers the keyword leg of the hybrid retriever.
# db.py — document_chunks, the searchable unit
CREATE TABLE IF NOT EXISTS document_chunks (
    id           BIGSERIAL PRIMARY KEY,
    document_id  BIGINT NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    chunk_index  INT NOT NULL,
    content      TEXT NOT NULL,
    embedding    vector({dim}) NOT NULL,   # dim = EMBEDDING_DIM (default 1024)
    metadata     JSONB DEFAULT '{}'::jsonb,
    UNIQUE (document_id, chunk_index)
);

Indexes & the dimension guard

Three indexes are created idempotently with CREATE INDEX IF NOT EXISTS — a plain b-tree on document_id (idx_chunks_doc) plus the two that make hybrid search fast:

IndexTypePowers
idx_chunks_embeddinghnsw (embedding vector_cosine_ops)Approximate nearest-neighbour vector search by cosine distance.
idx_chunks_content_tsvgin (content_tsv)Full-text ranking (ts_rank_cd) over the generated tsvector.
idx_chunks_docb-tree on document_idLooking up / cascading chunks per document.

After the DDL, ensure_schema() queries pg_attribute.atttypmod for the embedding column. If a pre-existing table's dimension doesn't match EMBEDDING_DIM, it raises rather than silently corrupting search.

⚠️ Switching embedding models is destructive

The vector column width is fixed at table-create time. If you change EMBEDDING_MODEL to one with a different dimensionality, ensure_schema() raises "document_chunks.embedding has dimension … but EMBEDDING_DIM is …". You must drop the table and re-ingest to switch models.

The connection pool

Connections come from a process-wide psycopg_pool.ConnectionPool, built lazily and guarded by a threading.Lock for thread-safe single initialisation. Pooling pays the "open TCP + auth + register adapter" cost once per pooled connection instead of on every query. 

# db.py — the lazily-built shared pool
_pool = ConnectionPool(
    conninfo=_database_url(),
    min_size=int(os.getenv("DB_POOL_MIN_SIZE", "1")),
    max_size=int(os.getenv("DB_POOL_MAX_SIZE", "10")),
    timeout=float(os.getenv("DB_POOL_TIMEOUT", "30")),
    max_idle=float(os.getenv("DB_POOL_MAX_IDLE", "300")),
    configure=_configure_connection,   # register_vector per connection
    kwargs={"autocommit": False},
    open=True,
)

true — master switch; set false to force direct connections
DB_POOL_MIN_SIZE

get_conn() & graceful fallback

Callers never touch the pool directly — they use the get_conn() context manager, which yields a pgvector-ready connection regardless of how it was obtained. The design degrades rather than crashes:

  • No psycopg_pool? The import is wrapped in try/except; if the optional dependency is missing, ConnectionPool is None and pooling is off.
  • Pool fails to build? An exception during construction prints "⚠️ DB pool unavailable, using direct connections", sets _pool_disabled, and never retries — it degrades, never crashes.
  • Pool off / unavailable? get_conn() opens a short-lived direct psycopg.connect(...), calls register_vector(conn), then commits on clean exit and rolls back on error.
# db.py — same API whether pooled or direct
@contextmanager
def get_conn() -> Iterator[psycopg.Connection]:
    pool = _get_pool()
    if pool is not None:
        with pool.connection() as conn:   # returned to the pool on exit
            yield conn
        return
    conn = psycopg.connect(_database_url(), autocommit=False)
    try:
        register_vector(conn)
        yield conn
        conn.commit()
    except Exception:
        conn.rollback(); raise
    finally:
        conn.close()

🪶 register_vector, once per connection

Pooled connections register the pgvector adapter via the configure=_configure_connection callback (run when each connection is created); direct connections call register_vector() inline. Either way vectors flow as native Python lists / numpy arrays with no manual casting. close_pool() tears the pool down on shutdown or in tests.

The embeddings factory

Embeddings live in one place — src/storage/embeddings.py — mirroring agents.llm. get_embedder() returns an OllamaEmbeddings client, cached with @lru_cache(maxsize=4) so repeated calls reuse the same client instead of reconstructing it. 

# embeddings.py — cached local embedder
@lru_cache(maxsize=4)
def get_embedder(model: str | None = None) -> OllamaEmbeddings:
    return OllamaEmbeddings(
        model=model or os.getenv("EMBEDDING_MODEL", "mxbai-embed-large"),
        base_url=os.getenv("OLLAMA_BASE_URL", "http://localhost:11434"),
    )

Two thin helpers wrap the cached client: embed_query(text) returns a single list[float], and embed_documents(texts) returns one vector per input. The default model mxbai-embed-large produces 1024-dimensional vectors, matching the EMBEDDING_DIM default in db.py.

📥 Pull the model before ingesting

The chosen model must exist in Ollama first — ollama pull mxbai-embed-large. Keep EMBEDDING_MODEL and EMBEDDING_DIM consistent: if they drift apart from the actual model output, the dimension guard in ensure_schema() will reject the mismatch.
