🔎 Retrieval Strategy

The retriever blends two complementary search methods — dense semantic vectors and sparse keyword matching — and fuses them with Reciprocal Rank Fusion. Optional LLM stages widen and sharpen recall when accuracy matters more than latency.

src/agents/vector_store.py src/agents/retrieval.py hybrid (default) multi-query · HyDE (opt-in)

The core idea: hybrid search

No single search method wins everywhere. Dense vector search understands meaning ("prepayment speed" ≈ "CPR") but can miss exact tokens; sparse full-text search nails exact identifiers (a CUSIP, a section number, "FNM 2026-012") but is blind to synonyms. Hybrid runs both and fuses the rankings, so the answer is grounded by whichever leg found the best evidence.

Two retrieval legs

🧲 Dense — semantic similarity

The query is embedded with mxbai-embed-large and compared to chunk vectors using pgvector's cosine distance operator <=>, accelerated by an HNSW index. 

# vector_search()
SELECT c.content, ..., 1 - (c.embedding <=> %s::vector) AS relevance_score
FROM document_chunks c JOIN documents d ON d.id = c.document_id
ORDER BY c.embedding <=> %s::vector LIMIT %s

🔤 Sparse — full-text keyword

Postgres' ts_rank_cd ranks chunks against a websearch_to_tsquery, so users can paste natural phrasing (quoted phrases, OR) without writing a tsquery. A GIN index over the stored content_tsv column makes it fast. 

# keyword_search()
WHERE c.content_tsv @@ websearch_to_tsquery('english', %s)
ORDER BY ts_rank_cd(c.content_tsv, websearch_to_tsquery('english', %s)) DESC

Fusion: Reciprocal Rank Fusion (RRF)

🔗 Why RRF instead of mixing scores?

Cosine similarity (0–1) and ts_rank_cd (unbounded) live on totally different scales — you can't just add them. RRF sidesteps this by ignoring raw scores and using only rank position: a chunk's contribution is 1 / (k + rank), summed across legs. It's parameter-light, robust to heterogeneous scorers, and rewards chunks that rank well in either leg (and especially in both). 

# _rrf_fuse() — over-fetch fetch_k=20 from each leg, then:
entry["rrf_score"] += 1.0 / (rrf_const + rank + 1)   # rrf_const = 60
# identity = (source_path, content[:80]) so the same chunk from both legs merges
merged = sorted(fused.values(), key=lambda x: x["rrf_score"], reverse=True)[:k]

Each leg over-fetches fetch_k=20 candidates; the fused list is trimmed to the final k=6 handed to the analyst. The retriever also de-duplicates on the first 100 characters of content before building the cited context block.

Optional query-side enhancements

These add LLM round-trips, so they're off by default to keep retrieval fast on CPU. Each is a single .env flag and degrades gracefully — a malformed LLM reply falls back to the plain query.

🪄 Conversational rewrite on by default

Turns a follow-up like "what about Q2?" into a standalone query using the last few chat turns, so a terse follow-up still retrieves correctly. Cheap (one short call on the fast model) and used for both routing and retrieval. retrieval.rewrite_conversational().

🌐 Multi-query expansion off

The planner LLM writes 3 paraphrases of the question; each is searched and the lists are RRF-fused. Widens recall for questions that can be phrased many ways. RETRIEVAL_MULTI_QUERY=true.

🧪 HyDE — Hypothetical Document Embeddings off

Generates a short, plausible answer to the question and embeds that alongside the raw query. An answer-shaped probe often sits closer in vector space to the real answer chunk than the question does. RETRIEVAL_HYDE=true.

🎯 LLM re-ranking off

Over-fetches, then an LLM scores each candidate 0–10 and keeps the most relevant top-k. The most expensive option — covered on its own page. Re-ranking strategy →
