📝 Prompt Optimization

Every agent runs on a small local model (1B–3B params). The prompts are engineered for that reality: tightly constrained outputs, explicit anti-hallucination guards, and parsers that survive a model going off-script.

Design principles

  • Constrain the output shape. Ask for exactly two words, or only a JSON array — small models ramble, so the format is pinned and the surrounding prose is forbidden.
  • Ground hard, hallucinate never. The analyst is told to answer only from the retrieved content and to cite specifics; with no context it must say so explicitly.
  • Match temperature to the job. Routing/classification run at temp 0.0 (deterministic); the analyst writes at temp 0.7 for fluency.
  • Budget tokens per role. The planner gets num_predict=24 (it only emits a label); the analyst gets 700 so answers don't run away on slow CPUs.
  • Always pair a prompt with a tolerant parser + a safety net. No prompt is trusted to be obeyed — keyword backstops and salvage parsers catch the misses.

The planner prompt — two labels, nothing else

The planner does double duty in one call: it decides route (document vs. general) and complexity (simple vs. complex, which drives the model router). The prompt demands exactly two words so the reply is trivial to parse. 

# graph.py — planner_node()
"Classify the user's request on two axes and reply with EXACTLY two "
"words separated by a space, nothing else:\n"
"  axis 1: DOCUMENT (...) or GENERAL (...)\n"
"  axis 2: SIMPLE (...) or COMPLEX (...)\n"
"Example replies: 'GENERAL SIMPLE' or 'DOCUMENT COMPLEX'."

🛟 Safety nets backstop the tiny classifier

Because a 1B model routes imperfectly, deterministic guards override it: a question containing obvious document words (_DOCUMENT_HINTS) forces retrieval when docs exist, and a long or analytical question (_COMPLEX_HINTS, >30 words) is promoted to complex even if the model said simple. The prompt sets the default; the code guarantees the floor.

The analyst prompt — grounded and anti-evasion

Built by the pure function analyst_prompt() (module-level so the eval harness can reproduce the exact app prompt without a database). It has two branches:

Context present?Instruction
Yes — chunks retrieved "Base your answer ONLY on the document content… Quote and cite specific details (numbers, names, sections). Do not claim you lack access — the content is provided here."
No — nothing retrieved "Answer using your general financial expertise. Be clear that the answer is not drawn from an uploaded document."

🚫 Killing the "I can't access your files" failure

Small instruct models love to refuse with "I don't have access to your documents" even when the text is right there in the prompt. The explicit "Do not claim you lack access" clause plus delimited === DOCUMENT CONTENT === fences fixes that, and forcing citations of concrete numbers/names anchors the answer to the evidence instead of the model's priors.

JSON-output prompts + salvage parsing

The rewrite, multi-query, HyDE, re-rank and question-suggestion prompts all ask for strict formats (a bare line, or a JSON array). Small models still drift, so each prompt is paired with a forgiving parser rather than trusting compliance.

  • Re-rank: asks for [{"id","score"}]; on any parse failure it returns the original order — the prompt can misfire without breaking retrieval.
  • Suggestions: _parse_question_list() tries strict JSON, then salvages quoted strings from a truncated array, then falls back to line-by-line bullet stripping.
  • Rewrite: strips stray quotes/markdown and takes only the first line, so a chatty model still yields a clean standalone query.

All clients are built through the single factory src/agents/llm.py (build_llm(num_predict, temperature, model), top_p=0.9), so these knobs live in one place.

Prompts are validated by evals, not vibes

The analyst prompt isn't tuned by guesswork — the eval harness runs each candidate model through the real analyst_prompt() and scores key-point coverage + an LLM judge, so prompt and model choices are measured together.
