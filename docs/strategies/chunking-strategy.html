✂️ Chunking Strategy

Documents are split into small, overlapping windows before embedding. The split size is deliberately tuned to the embedding model's context limit — not to a generic default.

src/agents/vector_store.py RecursiveCharacterTextSplitter 500 chars · 100 overlap

What it does

Every ingested file's text is divided into chunks of 500 characters with a 100-character overlap, using LangChain's RecursiveCharacterTextSplitter. Each chunk becomes one embedded row in document_chunks, the unit that retrieval later searches over. 

# src/agents/vector_store.py — VectorStore.__init__
# 500/100 keeps pathological char-heavy chunks (e.g. PDF ToC dot leaders)
# under mxbai-embed-large's 512-token context window.
self.text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=100,
)

Why these numbers

🎯 The binding constraint: the embedder's 512-token window

The embedding model is mxbai-embed-large, which encodes at most 512 tokens. Any chunk longer than that is silently truncated — the tail of the text never makes it into the vector. Financial PDFs are full of char-heavy content (table-of-contents dot leaders, number grids, tabular figures) where 1,000 characters can blow past 512 tokens. Capping at 500 characters guarantees every chunk fits, so no content is dropped at embed time.

  • Smaller chunks = sharper retrieval. A 500-char window is usually one fact, figure or clause, so a cosine match is precise rather than diluted across several unrelated sentences.
  • 100-char overlap preserves context across the seam. A sentence split between two chunks still appears whole in at least one of them, so boundary facts aren't lost.
  • "Recursive" splitting respects structure. The splitter tries paragraph breaks first, then lines, then sentences, then characters — keeping semantically related text together where it can rather than cutting mid-word.

The trade-off

Retrieval compensates for smaller chunks by returning several of them: the analyst sees the top RETRIEVAL_K=6 chunks together, so multi-chunk facts are reassembled in context.

♻️ Re-chunking is automatic on change

upsert_document() compares a SHA-256 content_hash. If a file is unchanged the embeddings are kept as-is; if it changed, its old chunks are deleted and the document is re-chunked and re-embedded from scratch — so tuning these numbers only takes effect for re-ingested content.

If you change the embedding model

The 500/100 figure is coupled to mxbai-embed-large. Swap the embedder and you should revisit it: a model with a larger window can take bigger chunks, and a different vector dimension requires updating EMBEDDING_DIM and rebuilding the document_chunks table.
