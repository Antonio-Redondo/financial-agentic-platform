Strategy · Performance

⚡ Semantic Response Cache

Generating an answer is the most expensive step in a query — a full LLM completion on CPU. When the same, or a very similar, question is asked again, this cache returns the previously generated answer instead of re-running the graph. It serves exact repeats in O(1) and near-duplicates by embedding similarity, while staying deliberately conservative so a close-but-wrong match never wins.

src/agents/semantic_cache.py exact O(1) + embedding similarity LRU + TTL

Two ways a query hits

get() tries the cheap path first, then falls back to embedding similarity. The query text is normalized — lower-cased and whitespace-collapsed via _normalize() — before either lookup, so trivial formatting differences still land an exact hit.

  • Exact hit — no embedding cost. The normalized query is keyed directly into an OrderedDict as (namespace, normalized_query). A direct dict.get means a verbatim repeat never calls Ollama at all.
  • Semantic hit — cosine similarity. On a miss, the query is embedded with the same local Ollama model used for retrieval, then dotted against every stored unit-normalized vector in the same namespace. The best match wins only if its similarity is ≥ RESPONSE_CACHE_THRESHOLD (default 0.97).
  • Returned payload. A hit returns a copy of the original process_query result with cached=True, cache_kind ("exact" / "semantic") and a rounded cache_similarity attached.

# semantic_cache.py — exact first, then similarity scan
norm = _normalize(query)
with self._lock:
    exact = self._store.get((namespace, norm))
    if exact is not None and not self._expired(exact):
        self._store.move_to_end((namespace, norm))   # LRU touch
        self.stats.hits_exact += 1
        return self._as_hit(exact, similarity=1.0, kind="exact")

Why the threshold is high

🧮 0.97 is on purpose

In finance, "CPR for pool A" and "CPR for pool B" embed very closely — but returning the wrong pool's answer is worse than a plain cache miss. The default RESPONSE_CACHE_THRESHOLD of 0.97 demands near-identical questions before a semantic match is served. Raise it to be stricter, lower it to trade safety for a higher hit rate; tune it for your corpus.

The similarity scan runs outside the lock — embedding is the slow part — and only the cheap bookkeeping (LRU touch, stat increments) is re-locked afterwards. If the embedder raises, the failure is swallowed and treated as a miss: the cache must never break a query.

Bounded LRU with optional TTL

The store is an OrderedDict used as an LRU: every read and write calls move_to_end(), so the least-recently-used entry is at the front. After each put(), _evict_if_needed() pops from the front until the store is back within RESPONSE_CACHE_MAX (default 256).

  • TTL is opt-in. RESPONSE_CACHE_TTL defaults to 0, meaning entries never expire. When set above zero, _expired() ages out entries by created_at on every read — expired entries are skipped on lookup rather than swept.
  • Thread-safe. All store mutations are guarded by an RLock, since Streamlit reruns execute on worker threads and a single process-wide singleton is shared across sessions.
  • Embed failures still cache. put() stores the entry even when the embedding comes back None — it just can't earn a semantic hit, only an exact one.
# eviction — drop least-recently-used until within RESPONSE_CACHE_MAX
def _evict_if_needed(self):
    cap = _max_size()
    while len(self._store) > cap:
        self._store.popitem(last=False)   # front == LRU
        self.stats.evictions += 1

Namespacing & self-invalidation

Every entry is scoped to a namespace built in FinancialAgent._cache_namespace() as "{model_part}|{corpus}" — the active model (or the literal "router" when the router is on) joined with the corpus signature from vector_store.chunk_count(). Both the exact key and the similarity scan are confined to a single namespace, so a cached answer can only ever surface under the exact conditions that produced it.

That makes the cache self-invalidating on the changes that would make an old answer wrong: switch the analyst model and the model_part changes; add, remove or re-chunk documents and the chunk count moves — either way the namespace shifts and the stale entries simply stop matching. On top of that, re-ingesting documents explicitly calls cache.clear() to drop everything. 

# financial_agent.py — namespace = model + corpus signature
def _cache_namespace(self) -> str:
    model_part = "router" if self.use_router else self.model
    try:
        corpus = self.vector_store.chunk_count()
    except Exception:
        corpus = "na"
    return f"{model_part}|{corpus}"
# on re-ingest, also wipe the cache outright
def rescan_documents(self):
    self.last_ingestion = ingest_folder(self.vector_store)
    self.cache.clear()   # new/changed chunks can change answers
    return self.last_ingestion

What bypasses the cache

⚠️ Correctness gating

The cache is consulted only for standalone questions. In FinancialAgent.process_query, cacheable = not history and not filters:

  • Follow-ups (history present) are rewritten using the conversation, so the query text alone doesn't identify the same answer.
  • Filtered searches (filters present) change retrieval, so the same text can legitimately produce a different answer.

In both cases the lookup is skipped and the result is never stored — the cache only ever holds answers that are safe to replay verbatim.

A cache hit short-circuits the whole graph: the planner, retriever and analyst never run. The cached answer is streamed straight to on_token and a trace line is prepended noting it was served from the semantic cache, before the payload is returned. PII redaction runs before the cache is touched, so the cache only ever sees already-scrubbed queries. 

# financial_agent.py — a hit skips plan / retrieve / generate
cacheable = not history and not filters
namespace = self._cache_namespace()
if cacheable:
    hit = self.cache.get(query, namespace)
    if hit is not None:
        if on_token:
            on_token(hit["output"])     # paint cached answer live
        return hit                    # graph never runs

🪶 Fails open, never breaks a query

get() returns None when the cache is disabled, the query is blank, or embedding fails — every path falls through to a normal graph run. A miss is always cheaper to recover from than a wrong hit, which is the whole design bias here.

Configuration knobs

RESPONSE_CACHE_ENABLED
true — master switch; set falsy to disable lookups and stores
RESPONSE_CACHE_THRESHOLD
0.97 — min cosine similarity for a semantic hit; higher = stricter
RESPONSE_CACHE_MAX
256 — LRU capacity; least-recently-used entries evicted past this
RESPONSE_CACHE_TTL
0 — seconds before an entry expires; 0 = never expire

📊 Observable behaviour

A CacheStats object tracks hits_exact, hits_semantic, misses, stores and evictions, and derives hits, lookups and hit_rate. Each served answer carries cache_kind and cache_similarity, and the UI surfaces a ⚡ cached marker on those messages.
