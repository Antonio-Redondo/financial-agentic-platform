🧭 Financial Agent

FinancialAgent is the single object the Streamlit UI talks to. It wires together the multi-agent graph, the guardrails, the semantic cache and the vector store, then exposes a small, stable interface — process_query(), set_model(), set_router() and a few stats helpers — while all the actual reasoning happens behind it.

src/agents/financial_agent.py runs the graph answer + trace + route + model

What the entry point does

FinancialAgent is a thin façade. Its job is to keep one steady public interface for the UI while delegating the real work to a planner → retriever → analyst graph, and to apply the cross-cutting concerns (guardrails, PII redaction, caching) that wrap every query.

On construction it builds the four collaborators, then does two startup chores before serving any query: it ingests the drop folder and warms up the model. 

class FinancialAgent:
    def __init__(self):
        self.vector_store = VectorStore()
        self.graph = FinancialGraph(self.vector_store)
        self.guardrails = get_guardrails()
        self.cache = semantic_cache.get_cache()
        # Ingest anything in the drop folder before serving queries.
        try:
            self.last_ingestion = ingest_folder(self.vector_store)
        except Exception as e:   # never block startup on ingest
            print(f"⚠️  Startup ingestion skipped: {e}", flush=True)
            self.last_ingestion = None
        # Preload the model so the first query skips cold-start cost.
        self.graph.warmup()

Module import calls load_dotenv(), so the .env configuration is in place before any collaborator is constructed.

The public API

Everything the UI needs is a method or property on this one class. The two properties below are read-only views onto the graph's current state; the setters mutate it in place and re-warm.

MemberKindRole
process_query(query, on_token=None, history=None, filters=None)methodThe main path. Runs guardrails → PII redaction → cache → graph and returns the response dict.
suggest_questions(history=None, n=4)methodProposes n follow-up questions grounded in the indexed docs and the user's recent questions. Best-effort — returns [] on error.
set_model(model)methodSwitches the active Ollama model in place (documents preserved), then warms it up.
set_router(enabled)methodToggles automatic model routing, then warms up the planner.
rescan_documents()methodRe-runs folder ingestion (the sidebar's manual trigger) and clears the response cache.
get_document_stats()methodReturns total_chunks, unique_documents and storage_type ("pgvector").
modelpropertyThe Ollama model currently in use (manual mode / routing off).
use_routerpropertyWhether automatic model routing is enabled.
documents_dirpropertyThe drop-folder path (from documents_dir()).
vector_storeattrThe VectorStore the UI reads directly for some stats.

♻️ Switch model, keep documents

set_model() and set_router() only touch the graph and then call warmup() — the indexed corpus in pgvector is untouched, so changing model mid-session doesn't re-ingest anything.

The return shape

process_query() always returns a Dict, but the keys differ by path. The success path carries the answer plus the trace, route and model that produced it: 

response = {
    "output": answer,                          # answer | analysis | "No analysis available."
    "trace": result.get("log", []),         # step-by-step log
    "route": result.get("route"),
    "complexity": result.get("complexity"),
    "analyst_model": result.get("analyst_model"),
    "search_query": result.get("search_query"),
    "cached": False,
}
  • Blocked path. If the input guardrail rejects the query, it returns early with output (the refusal reason), trace and "blocked": True.
  • Cache hit. Returns the stored response with a ⚡ Served from semantic cache line prepended to trace (the cached dict already carries cached: ...).
  • Error path. Any exception from the graph is swallowed into a friendly output message plus an empty trace — the UI never sees a stack trace.

How a query flows

The body of process_query() is an ordered pipeline. Each stage can short-circuit, so the graph only runs when the query is allowed, not cached, and (for caching) standalone.

Input guardrail

reject over-long / jailbreak input before any model runs

→
🙈
PII redact

scrub query + history before model, embedder, cache

→
⚡
Cache

standalone questions only — return on hit

→
🧠
Graph

planner → retriever → analyst, then cache the result

decision = self.guardrails.check_input(query)
if not decision.allowed:
    return {"output": decision.reason, "trace": [...], "blocked": True}

query = self.guardrails.redact_for_prompt(query)   # default-on PII scrub
history = self._redact_history(history)

cacheable = not history and not filters       # follow-ups / filters bypass cache
namespace = self._cache_namespace()
if cacheable:
    hit = self.cache.get(query, namespace)
    if hit is not None:
        ...                                       # paint + return cache hit

result = self.graph.run(query, on_token=on_token, history=history, filters=filters)

The cache namespace is built by _cache_namespace() from the model (or "router") plus the live chunk_count() — so cached answers self-invalidate when the model changes or the corpus grows/shrinks.

⚠️ Cache scope is deliberate

Only standalone questions are cached: a follow-up is rewritten using the conversation, and filters change retrieval, so the query text alone wouldn't identify the same answer. That's why cacheable = not history and not filters.

Startup ingestion & warmup

  • Folder ingestion. __init__ calls ingest_folder(self.vector_store) over DOCUMENTS_DIR (default ./data/documents), so dropping a file in and restarting is enough to index it. The call is wrapped in try/except — a failed ingest prints a warning and sets last_ingestion = None rather than blocking startup.
  • Model warmup. self.graph.warmup() preloads the model so the first real query doesn't pay the cold-start cost. It's re-run after every set_model() / set_router().
  • Manual rescan. rescan_documents() re-ingests on demand and then calls self.cache.clear(), because new or changed chunks can change answers.

📥 Drop-and-restart workflow

Because ingestion runs on construction, the supported flow is: copy documents into the drop folder, restart the app. For an in-session refresh, use the sidebar trigger that calls rescan_documents().
