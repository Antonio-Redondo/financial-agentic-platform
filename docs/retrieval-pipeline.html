Retrieval Pipeline — Multi-Query · HyDE · LLM Rerank

How the three optional retrieval stages work in the Financial Agentic Platform, and how they compose with hybrid (dense + sparse) search.

Source: src/agents/retrieval.py · Wired in src/agents/graph.py · Fusion helper in src/agents/vector_store.py

TL;DR. All three stages are opt-in env toggles. Multi-query asks the LLM for paraphrases; HyDE asks it to invent an answer-shaped probe; both are searched separately and fused with Reciprocal Rank Fusion. LLM rerank is a single-prompt cross-encoder that re-orders the top candidates. Defaults keep them off so retrieval stays fast on CPU.

2. k-budget — how the knobs interact

The graph chooses fetch sizes based on which stages are on, so it always over-fetches when fusing or reranking but never spends extra tokens when neither is active.

multi-query / HyDE rerank per-leg fetch returned to analyst
off off k (= 6) k
off on fetch_k (= 20) k after LLM rerank
on off fetch_k per variant k after RRF fuse
on on fetch_k per variant fetch_k after fuse → k after rerank

3. Multi-query expansion — expand_queries

Idea

Small dense models miss results whose phrasing differs from the query. Asking the LLM for paraphrases lets the same intent be retrieved from multiple angles.

Logic

  1. Prompt the planner LLM (self.planner_llm, the fast model) for a JSON array of n = 3 alternative phrasings.
  2. _extract_json is defensive: tries json.loads, then regex-finds the first [...] block, falls back to [] on parse failure.
  3. The original query is always prepended so you never lose the user's exact phrasing — variants only widen recall.
  4. Lowercase-dedupe + cap at n + 1.
Cost. Roughly 4× the search cost (one ANN + FTS per variant). Cost stays bounded because each variant fetches fetch_k = 20, and RRF compresses the union back to k = 6.

4. HyDE — hyde_query

Idea (Gao et al. 2022)

"Precise Zero-Shot Dense Retrieval without Relevance Labels." The gap between a question and a document chunk is large in embedding space (different syntax, different vocabulary). A hypothetical answer is structurally closer to the chunks you want to retrieve. So we let the LLM hallucinate a short answer-shaped passage and embed that as the search probe.

Logic

  1. Prompt the planner LLM to invent a 2–4 sentence "document excerpt" answer. Hallucination is explicitly encouraged — the text is never shown to the user, only used as a search probe.
  2. If anything goes wrong the function returns None and multi_query_search simply ignores it (original query carries the search).
  3. HyDE composes with multi-query. When both are on, the HyDE probe is appended as another variant alongside the paraphrases, all RRF-fused together.
Why it helps locally. With a 1024-dim embedder like mxbai-embed-large, a short answer phrased like a financial paragraph lands near the right chunks even when the user query is terse ("CPR vs PSA?").
Cost. One extra LLM round-trip + one extra retrieval leg per query.

5. Reciprocal Rank Fusion — _rrf_fuse

Multi-query and HyDE both produce multiple ranked lists that need merging. Naively summing scores does not work because the score scales are incompatible:

signalrangetypical magnitude
cosine similarity (pgvector)[0, 1]0.6 – 0.8
Postgres ts_rank_cdunbounded, small0.003 – 0.05
LLM judge score[0, 10]3 – 9

RRF sidesteps the problem by only using ranks:

score(d) = ΣL 1 / (k + rankL(d))

with k = 60 (the conventional smoothing constant).

Implementation details

  • Items are keyed by (source_path, content[:80]) so the same chunk surfaced by multiple legs gets its RRF scores summed.
  • The fused rrf_score overwrites relevance_score on the returned dict so downstream code (UI, analyst prompt) keeps working unchanged.
  • Truncated to k at the end.
Heads-up. Fused scores in the trace look like 0.032 instead of 0.69 — they are a different metric, not a broken cosine.

6. LLM cross-encoder rerank — rerank_llm

Idea

Dense ANN is recall-oriented and cheap, but ordering inside the top-k is noisy. A cross-encoder (one prompt that sees both query and passage together) is much more discriminating. We do not have a real cross-encoder model on the box, so we use the LLM-as-judge variant.

Logic

  1. Cap at 20 candidates — long prompts blow up CPU inference time and the model loses focus.
  2. Truncate each candidate to 600 chars and strip newlines so the batch fits in one prompt.
  3. Single prompt: "score each PASSAGE 0–10 for QUESTION, reply with JSON [{id, score}]".
  4. Defensive parsing again via _extract_json. If anything fails → return results[:top_k] (original order).
  5. Sort by score desc, take top_k, and renormalize the score to [0, 1] (score / 10) so the UI's relevance bar stays meaningful.
The key trick: one prompt for the whole batch. A per-passage prompt would be 20× slower. Trade-off: the model has to keep all 20 in working memory, which is why the 600-char cap matters.

7. Practical recipes

scenario setting why
Default — fast all off One ANN + one FTS, RRF-fused, top-6 to the analyst.
Hard, ambiguous questions RETRIEVAL_MULTI_QUERY=true Best recall boost per ms of extra compute.
Vocabulary mismatch (acronyms, short queries) RETRIEVAL_HYDE=true Cheapest after multi-query; closes the question/answer gap.
Precision over recall (near-misses in top-6) RETRIEVAL_RERANK=true Most expensive on CPU; highest-quality reordering.
Testing recall ceilings all on 5+ LLM calls per query on llama3.2:1b. Slow but thorough.
